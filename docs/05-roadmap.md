# 开发路线图

## 概述

obsidian-image-referer 的开发分为以下阶段，每个阶段有明确的交付物和验收标准。

## Phase 1 — MVP：基础 Referer 注入

**目标**：实现核心功能，用户可以配置全局 Referer 值并应用于所有图片请求。

**交付物**：

- 插件设置页面，包含 Referer 输入框
- 图片请求拦截器，注入自定义 Referer 头
- 配置持久化（loadData/saveData）
- 更新 manifest.json 为正式插件信息

**验收标准**：

- 在设置页面输入 Referer 值后，需要该 Referer 的图片能正常显示
- 插件禁用后图片加载恢复默认行为
- 不影响不需要 Referer 的图片正常加载
- 构建通过，无 TypeScript 类型错误

**预计时间**：1-2 天

---

## Phase 2 — 域名规则匹配

**目标**：支持按域名配置不同的 Referer 值，满足多图床场景。

**交付物**：

- 域名 → Referer 映射规则配置 UI
- 请求拦截器根据图片 URL 域名匹配对应 Referer
- 规则优先级：精确匹配 > 通配符匹配 > 全局默认值

**验收标准**：

- 不同域名的图片使用各自配置的 Referer
- 未匹配规则的域名使用全局默认 Referer
- 规则增删改即时生效

**预计时间**：2-3 天

---

## Phase 3 — 增强功能

**目标**：提升用户体验和灵活性。

**交付物**：

- 支持正则表达式匹配 URL 模式
- 按域名启用/禁用 Referer 注入的开关
- 图片加载失败时的诊断提示（可选）

**验收标准**：

- 正则规则能正确匹配目标 URL
- 禁用特定域名后该域名图片不注入 Referer
- 不引入明显性能开销

**预计时间**：3-5 天

---

## 里程碑总览

| 阶段 | 目标 | 关键交付物 | 状态 |
|------|------|------------|------|
| Phase 1 | 基础 Referer 注入 | 全局配置 + 请求拦截 | 未开始 |
| Phase 2 | 域名规则匹配 | 多规则配置 + 域名匹配 | 未开始 |
| Phase 3 | 增强功能 | 正则匹配 + 开关控制 | 未开始 |

## 测试要求

- 每个 Phase 完成后必须通过 `npm run build` 构建
- 在 Obsidian 中手动验证图片加载效果
- 验证插件启用/禁用的生命周期正确性
